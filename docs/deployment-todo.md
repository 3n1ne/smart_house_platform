# 部署 TODO

本清单只记录上线前后还需要补齐的部署、运维和生产验收工作。应用代码层面的核心功能、权限、日志、测试和基础文档已基本完成。

## P0 上线前必须完成

- [ ] 准备生产环境变量：复制 `backend/.env.example` 和 `frontend/.env.example`，填写生产数据库、密钥、上传目录、API 地址等配置。
- [ ] 后端启用生产模式：`FLASK_CONFIG=production`，`AUTO_CREATE_DB=false`，生产库建表只走 Alembic 迁移。
- [ ] 生成并托管强随机密钥：`SECRET_KEY`、`JWT_SECRET_KEY`、`SENSITIVE_DATA_KEY` 不得使用示例值，不得提交到仓库。
- [ ] 配置生产 MySQL：创建数据库和专用账号，限制账号权限，确认字符集、时区、连接数和备份权限。
- [ ] 执行数据库迁移：在 `backend/` 下运行 `flask --app run:app db upgrade`，并记录迁移版本。
- [ ] 初始化管理员账号：运行 `flask --app run:app seed-admin`，上线后立即修改初始密码。
- [ ] 构建前端：在 `frontend/` 下运行 `npm install` 和 `npm run build`，将 `dist` 交给静态资源服务。
- [ ] 配置反向代理：使用 Nginx 或同类服务托管前端，并代理 `/api`、`/uploads` 到后端。
- [ ] 启用 HTTPS：配置正式证书，开启 TLS 1.2/1.3，HTTP 自动跳转 HTTPS。
- [ ] 接入真实验证码通道：把当前动态验证码从“开发环境直接返回”改成短信、邮件或内部消息发送。
- [ ] 调整验证码存储：生产环境不要依赖单进程内存，改用 Redis 等共享存储，支持多进程部署和重启恢复。
- [ ] 关闭验证码明文返回：生产保持 `MFA_CODE_VISIBLE=false`。
- [ ] 配置上传目录：使用持久化目录或对象存储，禁止脚本执行，限制文件大小和可访问路径。
- [ ] 确认防火墙策略：公网只开放 80/443，数据库端口只允许后端访问。
- [ ] 配置后端进程托管：使用 systemd、Supervisor、Gunicorn/uWSGI 或等价方案运行 `backend/wsgi.py`。
- [ ] 配置日志：保留后端、Nginx、数据库日志，设置日志轮转和保留周期。
- [ ] 建立数据库备份：配置定时 `mysqldump` 或云数据库备份，并做一次恢复演练。
- [ ] 接入真实支付通道或明确关闭支付实收能力：避免演示支付被误认为真实支付。

## P1 上线验收

- [ ] 跑后端回归测试：`python -m pytest backend/tests`。
- [ ] 跑前端构建检查：`npm run build`。
- [ ] 跑安全验收脚本：`scripts/security_acceptance.ps1`，网络允许时不要跳过 `npm audit`。
- [ ] 跑性能验收：在生产等价环境执行压测，记录 P95 响应时间、失败率、CPU、内存、数据库连接数。
- [ ] 验证 2 秒响应目标：核心接口 P95 小于等于 2 秒。
- [ ] 验证 1000 并发目标：按需求记录并发压测报告，失败数应为 0 或给出明确整改结论。
- [ ] 做真实浏览器兼容验收：Chrome、Firefox、Edge、360 浏览器，以及 375、390、414、768 等移动端宽度。
- [ ] 人工验收核心流程：注册、登录、动态验证码、房源发布、房源筛选、预约、合同、支付、消息、维修、投诉、管理员报表。
- [ ] 检查权限边界：租客、房东、管理员互相不能访问越权数据或操作。
- [ ] 检查敏感信息：身份证号等敏感字段只脱敏展示，数据库中加密存储。
- [ ] 检查上传安全：非图片/视频、超大文件、脚本文件不能上传或执行。
- [ ] 检查生产错误页和接口错误：不能暴露堆栈、数据库连接串、密钥等内部信息。

## P2 上线后完善

- [ ] 接入监控告警：后端进程、接口错误率、响应时间、数据库连接、磁盘空间、证书到期。
- [ ] 接入集中日志：便于按用户、请求、错误码和操作日志追踪问题。
- [ ] 配置 WAF 或云安全策略：拦截常见 SQL 注入、XSS、扫描器和异常频率请求。
- [ ] 增加限流策略：登录、获取验证码、上传、消息发送等接口应限制频率。
- [ ] 完善验证码风控：加入发送冷却、每日上限、IP 限制、失败锁定和审计记录。
- [ ] 制定发布流程：灰度发布、回滚命令、数据库迁移回滚策略和发布检查表。
- [ ] 制定运维手册：服务启动、停止、重启、备份、恢复、排查日志、重置管理员密码。
- [ ] 定期做恢复演练：至少验证数据库备份和上传文件备份可以恢复。
- [ ] 定期漏洞扫描：保存依赖、主机和 Web 漏洞扫描报告。
- [ ] 明确数据保留策略：操作日志、消息、合同、支付记录、上传文件的保留和删除规则。

## 当前不建议上线前忽略的风险

- 当前动态验证码在开发环境会直接返回给前端，生产必须关闭并接入真实发送渠道。
- 当前验证码存储在后端进程内存中，多进程或重启场景不可靠，生产建议改 Redis。
- 支付能力如果没有真实渠道，只能作为流程演示，不能作为真实资金结算。
- 性能和兼容性必须在生产等价环境实测，本地脚本只能作为冒烟检查。
